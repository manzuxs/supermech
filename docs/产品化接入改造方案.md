# Supermech 产品化接入改造方案

> 说明：本文是“对外产品化接入”的规划文档，不是阶段 A 的当前执行基线。现阶段若与 `README.md` 或 `docs/当前实施重规划.md` 冲突，以后二者为准。

## 1. 目标

把当前仅适合在本仓库内运行的 `Supermech`，改造成一套可以稳定接入其他代码仓库的“可视化 Agent 工作台产品”。

目标不是简单复制 `apps/web` 到别的仓库，而是提供一套清晰的：

- 接入方式
- 包边界
- 配置协议
- 状态存储规范
- Agent 协作约定
- 版本升级路径

最终希望达到的效果：

1. 任意目标仓库只需要安装少量依赖和配置，即可启用可视化工作台。
2. Agent 只需要遵循统一的状态协议和技能约定，就能驱动 UI。
3. 前端、状态协议、文件监听、技能模板可以独立演进，不互相绑死。

---

## 2. 当前现状

当前仓库已经具备产品雏形，但本质上仍然是“单仓内实现”，不是“可复用产品”。

### 已具备的能力

- 有稳定的状态协议：`packages/schemas`
- 有本地文件监听与 API 中间层：`packages/watcher`
- 有可工作的前端工作台：`apps/web`
- 有三类技能工作流约定：`skills/`
- 有基本会话切换、技能切换、反馈回写、执行门禁能力

### 当前阻碍产品化的问题

#### 2.1 包边界还不清晰

- `watcher` 和 `schemas` 仍是 monorepo 内部私有包
- `apps/web` 直接依赖本仓库目录结构
- 技能文档与运行协议耦合，但没有单独发布策略

#### 2.2 存储模型命名混乱

当前存在三层概念，但边界不稳定：

- `plan`
- `skill`
- `session`

现实现状更接近：

- 一个 `plan` 对应一个工作主题目录
- 一个 `skill` 对应一个状态文件
- `sessionId` 曾常被写成 skill 名称，但当前基线要求它表示运行上下文，而不是 skill 身份

这会导致外部仓库接入时难以理解“该保存什么、切换什么、恢复什么”。

#### 2.3 接入方式仍然偏手工

要想在其他仓库使用，目前通常需要：

- 复制或搬运整个仓库结构
- 手动适配 `vite.config`
- 手动约定 `docs/supermech/` 目录
- 手动让 Agent 学会写对应 JSON

这还不是“产品接入”，而是“工程迁移”。

#### 2.4 文档与实现存在漂移

例如需求文档强调：

- 单一 `state.json`
- `/sessions` API
- 固定比例画布

但当前实现实际是：

- `.supermech/<plan>/state-<skill>.json`
- `/plans` 与 `/skills`
- 响应式自由画布

此外，`writing-plans` 中关于 `edges[]` 与 `metadata.dependencies` 的并行叙述在本文里只应视为待收敛问题记录，不应再被理解为正式双轨模型。

如果不先统一产品语义，外部接入后会持续混乱。

## 3. 产品化后的目标形态

建议把 `Supermech` 拆成四个可复用层。

### 3.1 `@supermech/schema`

职责：

- 定义 `WorkbenchState`
- 定义 `brainstorming / writing-plans / executing-plans` 的 metadata 类型
- 提供状态校验函数
- 提供版本号与迁移工具

要求：

- 独立发布
- 不依赖 React / Vite
- 能在 Agent 侧和前端侧同时使用

### 3.2 `@supermech/runtime`

职责：

- 文件存储读写
- 状态目录初始化
- plan/workspace/skill/session 管理
- 校验、升级、默认值补全
- 提供统一本地 API

建议包含：

- Node API
- 可选的 Vite 插件适配器
- 后续可扩展的 Express/Fastify 适配器

### 3.3 `@supermech/web`

职责：

- 提供现成 Workbench UI
- 提供可嵌入的 React 组件
- 支持主题、语言、布局配置
- 支持接收 runtime 提供的数据源

目标：

- 既能作为独立应用启动
- 也能嵌入目标仓库已有前端

### 3.4 `@supermech/skills`

职责：

- 提供标准技能模板与协议说明
- 规范 Agent 如何写状态、读反馈、切技能
- 提供最小示例状态文件和 prompt 约定

注意：

- 这里更像“协议与模板分发层”
- 不一定必须是 npm 包，也可以是模板仓库或脚手架资源

## 4. 推荐接入模式

产品化后，建议同时支持两种模式。

### 模式 A：Sidecar 模式

最适合当前阶段，也是第一优先级。

做法：

- 在目标仓库中安装 runtime + web
- 启动一个独立的 workbench 页面
- Agent 对目标仓库写状态文件
- workbench 监听并渲染

优点：

- 对目标仓库侵入最小
- 不依赖目标项目已有前端架构
- 最容易推广到不同技术栈仓库

适用场景：

- Node 服务
- Python 项目
- 多语言 monorepo
- 暂时不想改业务前端的团队

### 模式 B：Embedded 模式

做法：

- 目标仓库的现有 React 前端直接嵌入 Workbench 组件
- Runtime 作为本地服务或中间件提供状态

优点：

- 体验统一
- 更适合产品内部集成

缺点：

- 需要目标仓库本身有前端基础设施
- 接入复杂度明显更高

结论：

第一阶段只主推 Sidecar，等协议稳定后再做 Embedded。

## 5. 建议统一的核心语义

这是产品化改造里最应该先做的一步。

### 5.1 统一对象模型

建议明确：

- `workspace`
  - 一个被接入的代码仓库
- `session`
  - 围绕某个目标开展的一次完整工作上下文
- `skill`
  - 当前 UI/Agent 使用的工作模式，如 brainstorming、writing-plans、executing-plans

### 5.2 建议目录模型

不要继续使用 `plan` 和 `session` 混用。

建议改成：

```text
.supermech/
  config.json
  sessions/
    <session-id>/
      meta.json
      brainstorming.json
      writing-plans.json
      executing-plans.json
```

说明：

- `.supermech/` 明确是产品工作目录
- 一个 `session` 对应一组 skill 状态文件
- skill 文件名不再带 `state-` 前缀，减少噪音
- `meta.json` 专门存 session 级信息，避免塞进 skill 状态里

### 5.3 状态文件职责划分

建议：

- `meta.json`
  - session 标题
  - 创建时间
  - 最近活跃 skill
  - 目标仓库路径
  - schemaVersion
- `<skill>.json`
  - 该 skill 的画布状态
  - feedback
  - ui
  - skill-specific metadata

这样可以避免现在 `sessionId` 被写成 skill 名的歧义。

## 6. 推荐实施路线

按照“先稳协议，再抽 runtime，再包装接入”的顺序推进。

### 阶段 1：统一协议与命名

目标：

- 统一 `workspace / session / skill` 命名
- 冻结 v1 存储结构
- 给状态增加 `schemaVersion`

需要做的事：

1. 重写状态目录规范
2. 修正 `session-manager` 中默认状态生成逻辑
3. 更新前端和技能文档中的术语
4. 为旧目录结构提供兼容读取

验收：

- 新旧状态都能打开
- 新生成状态全部落到 `.supermech/sessions/...`

### 阶段 2：抽离 runtime

目标：

- 把 `watcher` 从“Vite 内部插件”升级为“产品运行时”

建议拆分：

- `storage`
- `session service`
- `state validator`
- `transport adapters`

子模块建议：

1. `storage/file-store.ts`
2. `services/session-service.ts`
3. `services/state-service.ts`
4. `adapters/vite.ts`
5. `adapters/http.ts`

验收：

- 不依赖 `apps/web` 也能单独初始化 session
- Vite 只是其中一个适配器，不再是唯一入口

### 阶段 3：把 web 包做成可嵌入产品

目标：

- 前端可以独立运行，也可以被宿主应用嵌入

建议改造：

1. `WorkbenchProvider` 不直接写死 `fetch('/__state/...')`
2. 抽出 `WorkbenchClient` 接口
3. 提供默认 `HttpWorkbenchClient`
4. UI 组件只依赖 client，不依赖固定 URL

验收：

- 在本仓库独立运行正常
- 在一个最小外部 React demo 中也能嵌入运行

### 阶段 4：提供官方脚手架

目标：

- 让外部仓库可以一条命令初始化

建议提供：

```bash
pnpm dlx create-supermech
```

初始化内容：

- 安装依赖
- 生成 `.supermech/config.json`
- 生成启动脚本
- 生成最小 session 示例
- 生成 Agent 使用说明

验收：

- 在全新仓库中 5 分钟内可启动第一个 workbench

### 阶段 5：发布兼容文档与升级机制

目标：

- 避免每次 schema 改动都让外部仓库手工救火

建议：

- `schemaVersion`
- `migrateState()` 工具
- 变更日志
- 升级指南

验收：

- 任意一次破坏性更新都有可执行升级路径

## 7. 推荐的外部仓库接入流程

这是最终希望交付给外部项目团队的标准流程。

### 7.1 初始化

在目标仓库执行：

1. 安装 `@supermech/runtime`
2. 安装 `@supermech/web`
3. 生成 `.supermech/config.json`
4. 生成 `pnpm supermech:dev` 启动命令

### 7.2 配置

最小配置建议：

```json
{
  "schemaVersion": 1,
  "workspaceRoot": ".",
  "storageDir": ".supermech",
  "defaultSkill": "brainstorming",
  "features": {
    "embedded": false,
    "qualityGates": true,
    "sessions": true
  }
}
```

### 7.3 启动

```bash
pnpm supermech:dev
```

运行结果：

- 启动 workbench
- 初始化默认 session
- 暴露本地状态 API
- 开始监听状态文件变化

### 7.4 Agent 协作

Agent 需要遵循固定约定：

1. 只写 schema 允许的 JSON
2. 通过 feedback 读取用户输入
3. 通过 activeSkill 控制界面
4. 在执行模式里只允许单任务 `active`

### 7.5 升级

升级时执行：

```bash
pnpm supermech:migrate
```

做的事情：

- 检查 schemaVersion
- 自动迁移目录和字段
- 输出迁移报告

## 8. 需要优先修复的设计问题

### P0：统一 session 模型

必须先解决：

- `sessionId` 语义不清
- `plan`/`session` 混用
- skill 文件和 session 文件职责混乱

不解决这个问题，后面所有产品化包装都会变成“给混乱结构套壳”。

### P0：把路径配置化

当前路径强依赖：

- `state.json`
- `docs/supermech`

产品化后必须支持：

- 自定义存储目录
- 自定义 session 根目录
- 自定义 API 前缀

### P1：把 transport 抽象出来

当前默认把运行时写进 Vite middleware，这对 React/Vite 友好，但对其他宿主不够通用。

必须支持：

- Vite
- 纯 Node HTTP
- 后续 Electron / desktop 容器

### P1：把前端请求层抽象出来

否则一旦外部仓库的 API 前缀、鉴权、宿主环境不同，`apps/web` 就不能复用。

### P2：补齐会话管理功能

当前 rename/delete 等能力在产品层定义了，但实现未闭环。

如果要对外使用，至少要补齐：

- rename
- delete
- duplicate
- import/export

## 9. 建议的验收标准

当以下标准成立时，可以认为进入“可接入其他仓库”的第一版。

### 产品级验收

1. 可以在一个全新 React 仓库中 5 分钟内启动
2. 可以在一个无前端的 Node 仓库中以 Sidecar 模式启动
3. 可以新建、切换、删除 session
4. Agent 写状态后，UI 能自动刷新
5. 用户反馈能稳定写回并被 Agent 读取
6. schema 升级有迁移工具

### 工程级验收

1. `schema` 可独立发布
2. `runtime` 可独立测试
3. `web` 可脱离本仓库运行
4. 不再依赖硬编码的 `docs/supermech`
5. 不再要求复制整个 monorepo 才能接入

## 10. 我建议的最小落地版本

如果只做一轮最有价值的改造，我建议收敛到下面这个 MVP。

### MVP 范围

- 把 `schemas` 改成正式 `schema` 包
- 把 `watcher` 升级成 `runtime` 包
- 改用 `.supermech/sessions/<session-id>/...` 存储结构
- `apps/web` 改成通过 client 接口访问 runtime
- 提供 Sidecar 启动命令
- 提供最小初始化脚手架

### 暂时不做

- 多用户协作
- 远程状态同步
- 数据库存储
- 真正的插件市场
- Embedded 深度定制能力

原因：

- 当前最关键的问题不是功能不够，而是产品边界不稳
- 先把单机、单仓、单 Agent 的产品链路做扎实，后续再扩展

## 11. 推荐执行顺序

### Step 1

重构存储语义与目录结构。

验证方式：

- 新建 session 后目录结构符合新规范
- 旧数据可兼容打开

### Step 2

抽离 runtime，并把 Vite 适配器挂到 runtime 下。

验证方式：

- 不启动前端也能通过 runtime API 读写 session

### Step 3

给前端引入 `WorkbenchClient` 抽象。

验证方式：

- 本仓库现有功能不回归
- 可以接一个 mock client 跑起来

### Step 4

提供 `create-supermech` 初始化工具。

验证方式：

- 在空仓库中可一键初始化并启动

### Step 5

补齐文档、升级工具和示例仓库。

验证方式：

- 新用户按文档可以独立完成接入

---

## 12. 结论

这个项目的核心方向是对的，尤其是：

- 以结构化状态为中介
- 把 Agent 工作流投影成 UI
- 通过反馈形成闭环

但它现在更像“仓库内工作台原型”，还不是“可被其他仓库稳定接入的产品”。

要把它产品化，最关键的不是继续堆新功能，而是先完成三件事：

1. 统一 `session` 语义
2. 抽离 `runtime`
3. 让 `web` 通过标准 client 接口消费 runtime

这三步完成后，`Supermech` 才真正具备对外接入价值。

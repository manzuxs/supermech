# 当前实施重规划

## 1. 文档目的

这份文档用于把当前仓库里分散的愿景文档、路线图、实施清单收敛成一份**可直接执行的当前计划**。

它不是替代所有旧文档，而是给后续开发提供一个明确基线：

- 先做什么
- 为什么先做
- 什么暂时不做
- 每个阶段如何验收

当前口径优先级：

1. `README.md`
2. 本文
3. 其他 `docs/*.md` 仅在不冲突时作为背景、路线图或历史记录引用

如果其他文档仍提到单 `state.json`、把 `sessionId` 当作 skill 名，或把 `edges[]` / `metadata.dependencies` 并列写成正式模型，应视为待清理的历史漂移，而不是当前基线。

---

## 2. 当前判断

基于当前代码、技能模板、README 和 `docs/` 目录内容，项目现状可以概括为：

1. 核心三技能工作台已经具备可用原型。
2. 主要问题不再是“有没有页面”，而是“协议、文档、技能模板、运行语义还没有完全收敛”。
3. 如果现在同时推进协议收敛、执行方法迁移、协作层产品化，对后续返工风险较高。

因此，后续推进顺序应调整为：

**先收敛底座，再补执行方法层，最后做协作层与产品化。**

---

## 3. 规划原则

本次重规划遵循以下原则：

### 3.1 协议优先于界面

优先统一状态语义、技能模板和运行约定，再决定界面如何承接。

### 3.2 主流程优先于扩展能力

优先把以下主链路做稳：

1. `brainstorming`
2. `writing-plans`
3. `executing-plans`

### 3.3 嵌入优先于新开画布

`subagent-driven-development`、`test-driven-development`、`systematic-debugging`、`requesting-code-review`、`receiving-code-review`、`verification-before-completion` 优先作为执行态能力嵌入，而不是单独新增主页面。

### 3.4 保留历史文档，但建立当前执行基线

旧文档继续保留，承担背景和设计演进记录的作用。  
这份文档负责定义当前最推荐的推进顺序。

---

## 4. 重新规划后的阶段

## 阶段 A：协议与文档收敛

### 目标

先消除当前最影响开发效率的漂移与歧义。

### 重点工作

1. 定稿 `sessionId`、`activeSkill`、`canvas.skillType` 的边界与含义。
2. 统一计划依赖表达，只保留 `edges[]` 或 `metadata.dependencies` 其中一种正式表达。
3. 确认唯一技能真相源，避免技能模板多份并行维护。
4. 清理旧文档中已失效的单 `state.json`、固定画布、旧目录结构叙述。
5. 收敛当前静态检查噪声，建立可持续的质量门槛。

### 暂不做

- 不新增第四个主画布
- 不引入新的执行方法技能页面
- 不做对外产品化接入改造

### 验收标准

1. `README`、`docs/`、schema、skill 模板对核心术语的说法一致。
2. `sessionId` 不再同时承担“技能名”和“运行实例标识”两种语义。
3. `writing-plans` 的依赖关系只保留一种正式表达。
4. `pnpm typecheck`、`pnpm build`、`pnpm check` 全通过。

---

## 阶段 B：把核心三技能闭环做稳

### 目标

把已有原型提升为稳定主流程，而不是只具备演示能力。

### 重点工作

1. 完成 `brainstorming` 的审批闭环。
2. 明确 `processedAt` 的展示与消费语义。
3. 稳定 `brainstorming -> writing-plans` 的阶段切换。
4. 稳定 `writing-plans -> executing-plans` 的 handoff。
5. 明确前端切 skill 时是读取已有状态还是生成默认状态。
6. 让“设计完成”“计划完成”“开始执行”这几个状态对用户可见。

### 暂不做

- 不引入完整 subagent 模型
- 不引入协作面板
- 不引入分支与 worktree 收尾流

### 验收标准

1. 三技能都能从空状态独立启动。
2. 三技能都能稳定读写 `feedback[]`。
3. 三技能切换时不丢关键状态。
4. `/execute` 不只是切页面，而是有清晰的执行入口语义（inline vs subagent）。
5. 用户能够看懂当前处于设计、计划还是执行阶段。

阶段 B 完成标志：`writing-plans` 暴露显式执行选择，`executing-plans` 从 `writing-plans` 水合生成，反馈处理状态在所有视图中可见。

---

## 阶段 C：把执行方法技能嵌入主流程

### 目标

让 `executing-plans` 真正承接 `superpowers` 的执行方法，而不是只展示任务卡。

### 重点工作

1. 为执行过程补充 `run` 语义。
2. 把 `subagent-driven-development` 嵌入执行态。
3. 把 `test-driven-development` 转化为任务步骤与执行门禁约束。
4. 把 `requesting-code-review` / `receiving-code-review` 转化为标准 review gate 与反馈流。
5. 把 `verification-before-completion` 转化为交付前检查。
6. 把 `systematic-debugging` 转化为可视化调查与回溯流。

### 暂不做

- 不单独新增 review 主画布
- 不单独新增 debugging 主画布
- 不做复杂多用户协作

### 验收标准

1. 执行视图能表达“实现 / 测试 / 评审 / 修复 / 重试”。
2. 用户能区分“任务做完”“任务通过 review”“任务待修复”。
3. 执行事件流可以承接 reviewer、implementer、gate 结果。
4. 新接入的执行方法能力不需要推翻已有三技能协议。

---

## 阶段 D：协作层与产品化

### 目标

在主流程稳定后，再做环境协作与对外接入，降低返工风险。

### 重点工作

1. 接入 `using-git-worktrees`、`dispatching-parallel-agents`、`finishing-a-development-branch`。
2. 增加分支、worktree、并行 agent 的上下文展示。
3. 补齐 plan / session 管理能力，例如 `rename`、`delete`、`duplicate`、`import/export`。
4. 抽象 runtime transport 层。
5. 抽象前端 client 请求层。
6. 设计迁移工具和外部仓库接入流程。

### 验收标准

1. 用户能看见当前工作所在的 branch / worktree。
2. 并行 agent 有独立状态与回收路径。
3. 收尾流程有明确的可视检查列表。
4. 新仓库能以最少步骤接入当前工作台。
5. 运行协议与前端渲染不再强耦合本仓库路径。

---

## 5. 当前最值得立刻开工的 5 件事

如果只做一轮高价值推进，建议按下面顺序开工：

1. 定稿 `sessionId` 语义，结束“它实际上只是 skill 名”的现状。
2. 统一 `writing-plans` 的依赖表达，去掉 `edges[]` 与 `metadata.dependencies` 双轨。
3. 清理文档漂移，把旧 `state.json` 主叙述降级为历史背景。
4. 把 `brainstorming` 的终态改成稳定进入 `writing-plans`，而不是直接回空画布。
5. 收敛 `pnpm check` 当前未通过的问题，建立阶段 A 的质量基线。

---

## 6. 明确暂缓项

为了避免扩线过早，以下内容暂缓到阶段 C 或 D：

- 单独的 review 页面
- 单独的 debugging 页面
- 多用户协作
- 远程同步
- 插件市场式能力分发
- 深度嵌入任意宿主的自定义前端集成

---

## 7. 与现有文档的关系

为了减少后续混淆，建议把现有文档按下面方式理解：

- `项目需求文档.md`
  - 作为最初愿景记录保留
  - 不再作为当前实施基线

- `Superpowers 可视化迁移整体计划.md`
  - 作为总方向和分层目标保留
  - 当前阶段以本重规划文档为实际执行顺序

- `Superpowers 体验对齐优化路线图.md`
  - 作为“体验差距清单”保留
  - 其中后半段应按阶段 C/D 执行，而非与阶段 A/B 并行推进

- `Phase 1 核心三技能协议收敛实施清单.md`
  - 作为阶段 A/B 的主要任务池
  - 但要以“先收敛高歧义项”为优先，而不是平均推进所有子项

- `产品化接入改造方案.md`
  - 作为阶段 D 的主参考
  - 不提前进入当前主线

---

## 8. 每阶段统一验证方式

每完成一轮阶段内改动，至少执行以下验证：

```bash
pnpm typecheck
pnpm build
pnpm check
```

另外补充一组工作流验证：

1. 新建 plan
2. 写入 `brainstorming` 状态
3. 切换到 `writing-plans`
4. 写入计划和依赖
5. 进入 `executing-plans`
6. 写入 `feedback / gate / event`
7. 重启后恢复状态

---

## 9. 最终目标

这次重规划后的目标不是“把所有设想一起做掉”，而是先把 `Supermech` 变成一个：

**拥有稳定主流程、统一协议、统一技能语义、可继续承接执行方法和协作能力的工作台底座。**

只有先做到这一点，后续接入 `subagent-driven-development`、review、debugging、productization 才会是加法，而不是返工。
